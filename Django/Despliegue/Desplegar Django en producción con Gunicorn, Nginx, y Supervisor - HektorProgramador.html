Desplegar Django en producción con Gunicorn, Nginx, y Supervisor

10 mayo, 201422 mayo, 2014 hektorDjango Framework / Python / Tutoriales1 Comment

Me lo han pedido varias veces ya, así que ahí va el tutorial.

Introducción

A continuación os resumo para qué sirve cada uno de los componentes que se utilizan en un despliegue genérico.

  • Gunicorn: Green Unicorn es un servidor WSGI HTTP para Python (pre-fork de unicorn de ruby). Consume poco y es bastante rápido.
  • Nginx: Es un servidor web/proxy inverso ligero de alto rendimiento y un proxy para protocolos de correo electrónico. Nos ayudará a servir ficheros estáticos.
  • Supervisor: Es un gestor de procesos para linux. Nos permitirá crear un proceso en segundo plano de nuestro servidor gunicorn.

Requisitos previos

– Virtualenv preparado con django funcionando en desarrollo
– Borrar del urls.py que django sirva los ficheros estáticos (si fuera el caso)
– Indispensable tener configurado correctamente el STATIC_ROOT en el settings.py:

import os
BASE_DIR = os.path.dirname(os.path.dirname(__file__))
STATIC_ROOT = os.path.join(BASE_DIR, 'static’)

Gunicorn

Instalamos gunicorn en el virtualenv:

pip install gunicorn

Lanzamos gunicorn manualmente desde la raiz (dónde está el manage.py):

gunicorn <proyecto>.wsgi:application --bind=127.0.0.1:8000 # se puede cambiar el puerto

Si entramos a http://127.0.0.1:8000 debería funcionar todo menos los ficheros estáticos. Si es así vamos bien.

Nginx

Instalamos nginx:

sudo apt-get install nginx
sudo service nginx start

Creamos un nuevo dominio en los sites-enabled (como en apache):

sudo nano /etc/nginx/sites-available/<proyecto>

Añadimos una configuración simple:

server {
        listen 8001;
        server_name 127.0.0.1;

        access_log off;

        location /static/ {
            alias /path/al/proyecto/static/;
        }

        location / {
            proxy_pass http://127.0.0.1:8000;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        }
}

Guardamos y creamos un enlace simbólico en sites-enabled:

cd /etc/nginx/sites-enabled
sudo ln -s ../sites-available/<proyecto>

Reiniciamos nginx:

sudo service nginx restart

Si entramos a http://127.0.0.1:8001 ya debería funcionar y cargar también los ficheros estáticos.

Básicamente definimos un proxy en 127.0.0.1:8001 del 127.0.0.1:8000 pero que también servirá los ficheros static, con lo que tendremos dos puertos ocupados. En producción real cambiaríamos el 8001 por el 80 y listos.

Sin embargo no es muy útil tener gunicorn en marcha manualmente, vamos a crear un pequeño demonio en segundo plano con supervisor.

Supervisor

Antes de continuar parad gunicorn si lo tenéis abierto manualmente.

Instalamos supervisor:

sudo apt-get install supervisor

Creamos un archivo de configuración del proyecto

sudo nano /etc/supervisor/conf.d/<proyecto>.conf

Añadimos una configuración simple:

[program:<proyecto>]
command = /path/al/proyecto/env/bin/python /path/al/proyecto/env/bin/gunicorn <proyecto>.wsgi:application
directory = /path/al/proyecto/<proyecto>
user = root

Es muy importante poner correctamente los paths al python y al gunicorn del virtualenv de nuestro proyecto. El usuario puede ser otro, pero debe tener los permisos adecuados.

Guardamos y activamos el proyecto

sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl start <proyecto>

Si ahora entramos de nuevo a http://127.0.0.1:8001 deberíamos tener funcionando el proyecto con gunicorn en segundo plano.

Para profundizar más sobre las opciones de configuración, como crear registros de errores y todo éso os dejo las documentaciones oficiales.
