Django con Nginx, Gunicorn, virtualenv, supervisor y postgresql

Posted on enero 29, 2014 by eliezerfot123

 
 
 
 
 
 
i
 
3 Votes


Django es un framework de desarrollo de aplicaciones web eficiente, versátil y evolucionando de forma dinámica. Cuando Django inicialmente ganó popularidad, la configuración recomendada para el funcionamiento de aplicaciones Django se basa en Apache con mod_wsgi. El arte de correr Django avanzó y en estos días la configuración recomendada es más eficiente y flexible, pero también más complejo e incluye herramientas tales como: Nginx, Gunicorn, virtualenv, supervisord y PostgreSQL.

En este texto voy a explicar cómo combinar todos estos componentes en un servidor Django en linux.

Lo primero es tener actualizado nuestro sistema

#aptitude update
#aptitude upgrade

POSTGRESQL

Para instalar PostgreSQL en un sistema basado en Debian ejecute este comando:


$ sudo apt-get install postgresql postgresql-contrib

Instalar virtualenv y crear un ambiente(environment) para la aplicación:

Virtualenv es una herramienta que te permite crear entornos Python separados en su sistema. Esto le permite ejecutar aplicaciones con diferentes conjuntos de requisitos concurrente (por ejemplo uno basado en Django 1.5, otro basado en 1.6). virtualenv es fácil de instalar en Debian:

$ sudo apt-get install python-virtualenv

Crear y activar un ambiente(environment) para su aplicación:

$ virtualenv --system-site-packages nombre_maquina_virtual

New python executable in nombre_maquina_virtual/bin/python
Installing distribute..............done.
Installing pip.....................done.

$ cd nombre_maquina_virtual
$ source bin/activate
(nombre_maquina_virtual) $

Su ambiente(environment esta activado y ahora puedes proceder a instalar django:

(nombre_maquina_virtual) $ pip install django

Su ambiente con django esta listo para su uso. Ahora crearemos un un proyecto de django nuevo: Anímate vamos empezando.. xD

(nombre_maquina_virtual) $ django-admin.py startproject hello

Tu puedes probar la aplicación en modo desarrollo:

(nombre_maquina_virtual) $ cd hello
(nombre_maquina_virtual) $ python manage.py runserver 127.0.0.1:8000

Configuramos postgresql para que trabaje con django

Para utilizar Django con PostgreSQL usted tendrá que instalar el adaptador de la base de datos psycopg2 en su entorno virtual. Este paso requiere la compilación de una extensión nativa (escrito en C). La compilación fallará si no puede encontrar los archivos de cabecera y las bibliotecas estáticas necesarias para vincular los programas en C con libpq (la biblioteca para la comunicación con Postgres) y módulos de construcción de Python (paquete python-dev). Tenemos que instalar estos dos paquetes primero, y luego podemos instalar psycopg2 usando PIP.

Instalamos algunas dependencias necesarias:


$ sudo apt-get install libpq-dev python-dev

Instalamos ahora el adaptador de la base de datos:

(nombre_maquina_virtual) $ pip install psycopg2

Tu puedes configurar la base de datos dentro del settings.py


DATABASES = {
'default': {
'ENGINE': 'django.db.backends.postgresql_psycopg2',
'NAME': 'nombre_base_de_datos',
'USER': 'tu_usuario',
'PASSWORD': 'arpacuatroymaraca',
'HOST': 'localhost',
'PORT': '', # Set to empty string for default.
}
}

Y finalmente podemos sincronizar ahora la base de datos para django:

(nombre_maquina_virtual) $ ./manage.py syncdb

GUNICORN

En la producción no va a utilizar el servidor de desarrollo de un único subproceso de Django, pero un servidor de aplicaciones dedicado llamados gunicorn.

Instale gunicorn en el entorno virtual de la aplicación:

(nombre_maquina_virtual) $ pip install gunicorn

Ahora que tenemos gunicorn instalado podemos probara que tal se comporta la aplicación con django ejecutando el siguiente comando:

(nombre_maquina_virtual) $ gunicorn hello.wsgi:application --bind 127.0.0.1:8001

Ahora deberíamos ser capaz de acceder al servidor de Gunicorn http://127.0.0.1:8001. He cambiado intencionalmente puerto 8000-8001 para forzar su navegador para establecer una nueva conexión.

Gunicorn está instalado y listo para servir a su aplicación. Estableceremos algunas opciones de configuración para que sea más útil. Me gusta establecer una serie de parámetros, así que vamos a poner a todos en un pequeño script BASH, que guardo como bin/ gunicorn_start o el nombre como diríamos en Maracaibo (COMO VOS QUERAIS)


#!/bin/bash

NAME="hello" # Nombre de la aplicación
DJANGODIR=/direcion/de/la/aplicacion/hello # Ubicación de donde esta tu proyecto django
SOCKFILE=/direcion/de/la/aplicacion/run/gunicorn.sock # Nos comunicaremos usando unix socket
USER=usuario # Usuario que tiene los superpoderes
GROUP=grupo_del_usuario # El grupo al que pertenece
NUM_WORKERS=3 # Cuantos procesos debería trabajar Gunicorn spawn
DJANGO_SETTINGS_MODULE=hello.settings # El archivo de configuración donde esta
DJANGO_WSGI_MODULE=hello.wsgi # WSGI module name

# Activamos el entorno virtual
cd $DJANGODIR
source ../bin/activate
export DJANGO_SETTINGS_MODULE=$DJANGO_SETTINGS_MODULE
export PYTHONPATH=$DJANGODIR:$PYTHONPATH

# Creando el directorio run si por casualidad no existe.
RUNDIR=$(dirname $SOCKFILE)
test -d $RUNDIR || mkdir -p $RUNDIR

# Corremos nuestra aplicación django
# Programs meant to be run under supervisor should not daemonize themselves (do not use --daemon)
exec ../bin/gunicorn ${DJANGO_WSGI_MODULE}:application \
--name $NAME \
--workers $NUM_WORKERS \
--user=$USER --group=$GROUP \
--log-level=debug \
--bind=unix:$SOCKFILE

gunicorn_start será capaz de corre la aplicación con el usuario. Nosotros deberíamos tener propiedad de acceder al directorio con el usuario que hemos asignado.

Ahora comenzaremos el monitoreo con SUPERVISOR:

Su guión gunicorn_start debe estar listo y funcionando. Tenemos que asegurarnos de que se inicie automáticamente con el sistema y que se puede reiniciar automáticamente si por alguna razón se cierra inesperadamente. Estas tareas pueden ser fácilmente manejadas por un servicio llamado supervisord. La instalación es simple:

$ sudo apt-get install supervisor

Cuando se instala Supervisor le puedes dar para iniciar programas y ver mediante la creación de archivos de configuración en el directorio /etc/supervisor/conf.d. Para nuestra aplicación hello vamos a crear un archivo llamado /etc/supervisor​/conf.d/ hello.conf con este contenido:


[program:hello]
command = /direccion/donde/esta/la/maquina/virtual/bin/gunicorn_start ; Comando para iniciar la aplicación
user = usuario ; Usuario que corre la aplicación
stdout_logfile = /donde/quieres/guardar/los/logs/gunicorn_supervisor.log ; Donde se escribira la aplicación
redirect_stderr = true
autostart=true
autorestart=true

Puede configurar muchas otras opciones, pero esta configuración básica debe ser suficiente.

Cree el archivo para almacenar los mensajes de registro de la aplicación:


$ mkdir -p /donde/se/guardaron/los/logs/
$ touch /donde/pusistes/los/logs/gunicorn_supervisor.log

Después de guardar el archivo de configuración para el programa usted puede pedir supervisor para releer archivos de configuración y actualización (que se iniciará la aplicación).


# supervisorctl reread
hello: available
#supervisorctl update
hello: added process group

TU puedes entonces ver el estatus de tu aplicación o iniciarla, detenerla, reiniciarla con el uso del supervisor. (YA ESTAMOS CASI LISTOS PARA LA DIVERSIÓN)

$ sudo supervisorctl status hello
hello RUNNING pid 18020, uptime 0:00:50
$ sudo supervisorctl stop hello
hello: stopped
$ sudo supervisorctl start hello
hello: started
$ sudo supervisorctl restart hello
hello: stopped
hello: started

Su aplicación ahora debe iniciarse automáticamente después de un reinicio del sistema y se reinicia automáticamente si es que alguna vez se estrelló por alguna razón.

AHORA CONFIGURAREMOS NGINX

Ahora disponemos el tiempo para configurar Nginx como un servidor para la aplicación y sus archivos estáticos. Instale e inicie Nginx:


$ sudo apt-get install nginx
$ sudo service nginx start

Ahora podemos navegar en su servidor http://localhost/ y sencillamente tendremos (“Welcome to nginx!”).

Crearemos una configuración virtual para django en nginx

(OHHH ES DIFICIL?) No, no es difícil:

Cada servidor virtual Nginx debe ser descrito por un archivo en el directorio /etc/nginx/sites-available. Usted habilita los enlances simbolicos a el directorio /etc/nginx/sites-enable/

Crear un nuevo archivo de configuración del servidor nginx para su aplicación Django ejecuta en localhost en /etc/nginx/sites-available/default. El archivo debe contener algo parecido a lo siguiente:


upstream hello_app_server {
# fail_timeout=0 means we always retry an upstream even if it failed
# to return a good HTTP response (in case the Unicorn master nukes a
# single worker for timing out).

server unix:/webapps/hello_django/run/gunicorn.sock fail_timeout=0;
}

server {

listen 80;
server_name example.com;

client_max_body_size 4G;

access_log /webapps/hello_django/logs/nginx-access.log;
error_log /webapps/hello_django/logs/nginx-error.log;

location /static/ {
alias /webapps/hello_django/static/;
}

location /media/ {
alias /webapps/hello_django/media/;
}

location / {
# an HTTP header important enough to have its own Wikipedia entry:
# http://en.wikipedia.org/wiki/X-Forwarded-For
proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;

# enable this if and only if you use HTTPS, this helps Rack
# set the proper protocol for doing redirects:
# proxy_set_header X-Forwarded-Proto https;

# pass the Host: header from the client right along so redirects
# can be set properly within the Rack application
proxy_set_header Host $http_host;

# we don't want nginx trying to do something clever with
# redirects, we set the Host: header above already.
proxy_redirect off;

# set "proxy_buffering off" *only* for Rainbows! when doing
# Comet/long-poll stuff. It's also safe to set if you're
# using only serving fast clients with Unicorn + nginx.
# Otherwise you _want_ nginx to buffer responses to slow
# clients, really.
# proxy_buffering off;

# Try to serve static files from nginx, no point in making an
# *application* server like Unicorn/Rainbows! serve static files.
if (!-f $request_filename) {
proxy_pass http://hello_app_server;
break;
}
}

# Error pages
error_page 500 502 503 504 /500.html;
location = /500.html {
root /webapps/hello_django/static/;
}
}

Luego de configurar nuestro archivo con los datos pertenecientes entonces podemos hacer el enlace simbólico

$ sudo ln -s /etc/nginx/sites-available/default /etc/nginx/sites-enabled/default

Ahora reiniciamos nuestro nginx

# service nginx restart

Si se desplaza a su sitio, usted debe ahora ver su bienvenida de las páginas de Django funciona con Nginx y Gunicorn. Ahora es momento de seguir adelante y desarrollar el contenido que desees.

A la fina podriamos tener un directorio parecido a esto. Aunque usted puede configurarlo y establecerlo como mejor le parezca:

/webapps/hello_django/
├── bin <= Directory created by virtualenv
│ ├── activate <= Environment activation script
│ ├── django-admin.py
│ ├── gunicorn
│ ├── gunicorn_django
│ ├── gunicorn_start <= Script to start application with Gunicorn
│ └── python
├── hello <= Django project directory, add this to PYTHONPATH
│ ├── manage.py
│ ├── project_application_1
│ ├── project_application_2
│ └── hello <= Project settings directory
│ ├── __init__.py
│ ├── settings.py <= hello.settings - settings module Gunicorn will use
│ ├── urls.py
│ └── wsgi.py /usr/include/python2.7
├── lib
│ └── python2.7
├── lib64 -> /webapps/hello_django/lib
├── logs <= Application logs directory
│ ├── gunicorn_supervisor.log
│ ├── nginx-access.log
│ └── nginx-error.log
├── media <= User uploaded files folder
├── run
│ └── gunicorn.sock
└── static <= Collect and serve static files from here
